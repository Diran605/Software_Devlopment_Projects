1. Clock-In / Clock-Out (Attendance)
The system needs to verify that a worker was actually present before accepting any plans or activity logs. This is solved with a clock-in/clock-out mechanism that acts as the container for a worker's entire day.
1.1 How it works
When a worker logs in, their dashboard is visible but daily plans and timers are locked until they clock in.
Clicking Clock In creates a work session for that day and unlocks the full portal.
Clicking Clock Out stops all running timers, triggers the plan carry-over prompt, seals the day, and locks the portal again.
One session per day per worker — enforced at both database and API level.
1.2 What clock-in captures
Exact timestamp of clock-in.
IP address and browser/device info — so admins can spot unusual login locations.
A unique work session ID that all plans and logs for that day are linked to.
1.3 Clock-Out behaviour
System checks for any running or paused timers — shows confirmation prompt to stop and save them.
After timers are resolved, system checks for any incomplete plans and shows the carry-over prompt (see Section 8).
After both are resolved, clock-out completes and total minutes for the day are stored on the session.
1.4 Abandoned Sessions (Forgot to Clock Out)
A scheduled Laravel job runs at a configurable time (e.g. 8pm or X hours after standard closing time).
It finds all sessions still marked active with no clock-out.
It closes them automatically, sets status to system_closed, and stops any running timers.
Incomplete plans are also flagged — the worker sees the carry-over prompt the next time they log in.
Worker receives an inbox notification informing them their session was auto-closed.
Admin is also notified and can see all flagged sessions in the Attendance module.
1.5 Worker Session Reopen Request
When a worker's session gets auto-closed and they need to continue working, they send a formal request through the inbox rather than reopening it themselves.
Worker sees a notice on their dashboard: "Your session for today was automatically closed. If you need to continue, request a reopen from your admin."
They click Request Reopen — a reason is mandatory before submitting.
Submitting creates a reopen request record and automatically sends a message to the admin's inbox with Approve and Decline buttons inline.
If Approved: session flips back to active, worker gets an inbox notification, portal unlocks, approval written to audit log.
If Declined: worker gets an inbox notification with the admin's optional decline reason. Session stays closed.
Admin can also reopen a session directly from the Attendance module without waiting for a worker request — for cases where the admin notices the issue proactively.

1.7 Attendance Module in the Admin Panel
Attendance records are visible in a dedicated Attendance module inside the Filament Admin Panel.
What the Attendance module shows:
A table of all workers with clock-in time, clock-out time, total hours, and session status for any selected day or date range.
Status indicators — green for normal closed sessions, amber for system_closed (auto-closed), red for days with no session at all (absent).
Carried-over plan count per worker per day — so admin can see at a glance if someone has a growing backlog.
Ability to drill into a worker's session to see all their plans, logs, and any carry-overs for that day.
Reopen Session action accessible directly from this view.
Pending reopen requests appear at the top as a priority list requiring action.
Export attendance records to PDF on company letterhead.
Access scope:
Admin sees their department only.
Super Admin sees all departments and all workers.

2. Activity Timer on Daily Plans
Workers can start a live timer directly from a daily plan task. When stopped, the timer automatically creates an activity log entry — no manual re-entry needed.
2.1 Timer States
State
Description
How to get here
Idle
Timer not started
Default state of every plan
Running
Timer actively counting
Worker clicks Start
Paused
Counting stopped, time saved
Worker clicks Pause
Stopped
Complete, log created
Worker clicks Stop or auto-stop at clock-out
2.2 State Transitions
Idle → Running: worker hits Start.
Running → Paused: worker hits Pause.
Paused → Running: worker hits Resume.
Running → Stopped: worker hits Stop, or clock-out triggers auto-stop.
Paused → Stopped: clock-out auto-stops even paused timers, capturing accumulated time.
2.3 Auto-Stop at Clock-Out
At clock-out, system checks for any Running or Paused timers.
Shows confirmation prompt with task name and accumulated time.
On confirm — timer stops, activity log created with stop_reason = clock_out.
2.4 Abandoned Timer Safety Net
A scheduled job runs every 30 minutes.
Finds timers in running state where timer_started_at is older than the configured threshold (default 3 hours, adjustable per organization by super admin).
Auto-stops the timer, calculates duration up to cutoff, creates activity log with stop_reason = system_timeout.
Sets is_verified = false on the log entry so admin can review it.
Notifies both the worker and their admin via the inbox.

3. Admin-Assigned Recurring Tasks
Admins can create task templates that automatically appear in workers' daily plans on a set schedule — for example a weekly report every Monday or a daily standup note every day.
3.1 How it works
Admin creates a task template with a name, work type, expected duration, and recurrence schedule.
They assign it to specific workers, the whole department, or everyone.
A Laravel scheduled job runs every midnight and auto-generates daily_plans rows for all assigned workers on the days the template is due.
Workers see the task pre-populated in their plan. They cannot delete it since it was assigned, but they log against it and mark completion as normal.
Admin can see compliance — who completed the assigned task and who did not.

4. Worker Personal Recurring Tasks
Workers can set up their own recurring tasks independently — without any admin involvement. These are personal habits or routines the worker wants auto-generated in their daily plan on a schedule they define themselves.

4.2 How it works
Worker goes to their daily plan section and creates a personal recurring task — giving it a name, work type, priority, expected duration, and recurrence schedule (daily or weekly on a specific day).
The same midnight scheduled job that handles admin templates also picks up active personal recurring tasks and auto-generates plan entries for the worker on the scheduled days.
The worker can edit or delete their personal recurring task at any time.
Deleting a personal recurring task stops future auto-generation but does not delete already-generated plan entries.
Admin can see personal recurring task entries in the team overview just like any other plan — they appear as normal plan entries with no special admin controls over them.

5. Daily Plan Carry-Over
When a worker ends their day with incomplete plans, the system always prompts them to decide what to do with each one. Nothing is carried over automatically — the worker makes the decision themselves every time.
5.1 When the prompt appears
At clock-out — after timers are resolved, before the session closes, the system checks for any plans still in pending or partial status.
If any are found, the carry-over prompt appears. Clock-out does not complete until the worker has made a decision on every incomplete plan.
If the session was auto-closed by the scheduled job (worker forgot to clock out), the carry-over prompt appears the next time the worker logs in — shown as a priority notice before they can clock in for the new day.
5.2 The carry-over prompt
Each incomplete plan is shown individually with the following:
Task name — read only.
Completion status — shows whether the task was not started or partially done (based on linked activity logs).
Priority — pre-filled with yesterday's priority but editable. Worker can upgrade to High if the task is now overdue and more urgent.
Three action buttons — Carry Over, Cancel, Leave for now.
The worker decides per task — not a blanket carry-all button. Each task gets its own decision.
5.3 The three choices explained
Choice
What happens
Plan status after
Carry Over
A new plan entry is created for tomorrow. Priority is saved as adjusted. carried_from_plan_id links back to today's plan. Original plan marked carried_over.
Today: carried_over. Tomorrow: pending (new entry)
Cancel
Plan is marked cancelled. No carry-over. No new entry created.
Today: cancelled
Leave for now
Plan stays as pending on today's date. Surfaces in a 'Pending from previous days' section on the worker's dashboard.
Today: pending (unchanged)
5.4 Priority adjustment at carry-over
When a worker chooses Carry Over, the priority field is pre-filled with the original plan's priority.
The worker can change it before confirming — for example upgrading a Medium priority task to High because it is now overdue.
The adjusted priority is saved on the new carry-over plan entry. The original plan keeps its original priority unchanged.
5.5 Carry-over chain tracking
Every time a plan is carried over, the new entry stores carried_from_plan_id pointing to the plan it came from.
carry_over_count increments by one each time — so a task carried over 3 days in a row has carry_over_count = 3.
This chain allows full traceability — admin can trace a task back to when it was originally created and see how many times it has been rolled over.
5.6 Admin visibility of carried-over plans
In the team overview, carried-over plan entries show an amber 'Carried Over' badge next to the task name.
Admin can filter the team overview by carry-over status to quickly spot workers with a growing backlog.
If the same task has a carry_over_count above the configured threshold (default 3, adjustable by admin), it automatically appears in the admin's attention list as a flag — not a disciplinary alert, just a prompt to check in with the worker.
The carry-over count and history are visible when the admin drills into a worker's plan detail.

7. Company Letterhead
Organizations upload their official letterhead as separate header and footer images. The system automatically detects dimensions — no manual pixel input required from the admin.
7.1 Upload and setup flow
Admin uploads a header image (PNG recommended, transparent background supported).
Admin uploads a footer image separately.
System automatically reads the natural dimensions of both images using PHP's getimagesize() — no height input needed from the admin.
A live preview renders immediately showing exactly how a sample letter will look.
Admin sets an accent color used for tables, dividers, and headings inside the letter body.
Only one active letterhead per organization at a time.

8. Letters Module in the Admin Panel
The Letters module lives inside the Filament Admin Panel and has four sub-sections covering everything related to official company correspondence.
8.1 Letterhead Settings
Admin uploads header and footer images here.
Live preview renders automatically after upload — no manual configuration needed.
Admin sets accent color for letter body content.
Shows current active letterhead with upload date.
Previous letterheads are kept in history but only one is active at a time.
8.2 Letter Templates
Lists all the company's templates — appointment, warning, query, confirmation, and custom ones.
Shows which are based on system defaults and which have been customized.
Admin edits templates in a rich text editor with a sidebar of available placeholders to click and insert.
Admin can create new custom templates from scratch.
Templates are per-company — edits never affect other organizations or the system defaults.
8.3 Generate Letter
Admin selects a worker from a dropdown and selects the letter type.
System loads the company's customized template and auto-fills all system placeholders.
Admin fills in any custom fields specific to that letter.
Live preview renders on screen showing the actual header image, letter body, and footer image exactly as it will print.
Admin clicks Generate — PDF is created using Browsershot rendering a Blade view.
PDF is stored, recorded in generated_letters, and automatically delivered to the worker's inbox as a letter-type message with the PDF attached.
If the letter type requires acknowledgement (warning, query), the worker sees an acknowledgement prompt in their inbox.
8.4 Issued Letters
Full history of every letter ever generated — filterable by worker, letter type, and date range.
Shows acknowledgement status — pending or confirmed — for letters that require it.
Admin can re-download or resend any previously issued letter.
Super Admin can see issued letters across all organizations.

8.8 Worker Acknowledgement
Worker receives the letter in their inbox as a message with the PDF attached.
If acknowledgement is required, a prompt appears: "This letter requires your acknowledgement. Click below to confirm you have read it."
Worker clicks Acknowledge Receipt — acknowledged_at and acknowledged_by are recorded immediately.
Admin sees acknowledgement status in the Issued Letters view — pending or confirmed.
Creates a legally traceable record entirely within the system — no paper signatures needed.
